## 0. 一句话目标

在 **DeepTutor 仓库内 fork 改造**，套一层「课程」外壳：学生在主页选一门课（首版只做
「高等数学」）→ AI 做一段 **onboarding 对话**问清情况 → 系统据此**裁剪一份预设的
「精通之路」模板** → 进入 DeepTutor **原生精通之路 UI** 学习。核心卖点是每门课
**内置好 RAG 资料，即开即用**。原生那堆功能页不删，只用一个**开关**把导航入口藏起来
（它们还要当"备课台"用）。

## 1. 背景与关键判断（为什么这么做）

DeepTutor 的「精通之路」不是一个独立状态机页面，而是**对话式导师 loop + 一组工具**
（`mastery_status/quiz/grade/assess/build`）。一条学习路径就是一份 JSON
（`LearningProgress`），存成 `{book_id}.json`。这带来两个决定性事实：

1. **"预设课程模板"技术上就是一份预先做好的 `LearningProgress` JSON**。裁剪 = 加载它、
   删/排模块、改起点，再换个 `book_id` 存盘。不用碰学习引擎内核。
2. **RAG 是"具名知识库(KB)"制**，一门课 = 一个 KB，检索天然按课隔离。

因此整套思路是干净成立的，改造集中在"外壳"（主页/onboarding/预览/开关）+ 一个把
profile 落到模板上的**裁剪服务**，DeepTutor 的引擎、RAG、WebSocket、学习 UI 全部复用。

## 3. 现有代码锚点（已亲自核实，可直接对照）

### 3.1 学习引擎 / 数据模型（后端，几乎不用改）

- **路径 JSON 的 schema** = `LearningProgress`：`DeepTutor/deeptutor/learning/models.py:185`
  - 关键字段：`book_id`、`modules: list[LearningModule]`、`current_module_id`、
    `current_stage`、`mastery_levels`、`qualitative_mastery`、`knowledge_types`、
    `current_kp_index`、`diagnostic`。
  - `LearningModule`(`models.py:89`)：`id / name / order / pass_threshold(默认0.7) /
    knowledge_points`。
  - `KnowledgePoint`(`models.py:80`)：`id / name / type / module_id`。
    `type` ∈ `KnowledgeType`(`models.py:24`) = `memory | concept | procedure | design`。
    （定量门槛作用于 memory/procedure，费曼定性门槛作用于 concept/design。）
- **存储层**（纯 JSON 读写，就地改字段完全可行）：`DeepTutor/deeptutor/learning/storage.py`
  - `LearningStore.save(progress)` / `.load(book_id)` / `.exists` / `.list_all` /
    `.delete`。存储目录 = `workspace/learning/{book_id}.json`
    （`storage.py:30` 由 `get_path_service().get_workspace_dir()` 决定）。
  - `book_id` 有安全校验：不能含 `/ \ .. :`（`storage.py:34`）。裁剪产物的 book_id 要遵守。
- **门槛/地图策略**：`DeepTutor/deeptutor/learning/policy.py`
  - `next_objective(progress)`、`map_summary(progress)` —— 预览页数据就来自这里。
- **service**：`DeepTutor/deeptutor/learning/service.py`（`LearningService`：
  `get_or_create / init_modules / save / list_progress`）。

### 3.2 学习 REST API（已存在，直接调用即可注入模板 & 取预览）

### 3.3 精通之路对话能力（把 profile-path 接进对话的关键）

### 4.2 onboarding 对话 + 裁剪服务（决策5/6/7）—— 核心

**4.2.1 后端：profile 收集 + 模板裁剪，新增一个课程外壳路由**

新增 `DeepTutor/deeptutor/api/routers/course_shell.py`，注册前缀 `/api/v1/course`
（在 `main.py` 仿照 `mastery_path` 加一行 `include_router`）。端点建议：

- `GET  /courses` — 返回课程目录（首版硬编码或读一个 `courses.json`，
  见 §5）。每门课含：`id / title / summary / difficulty / duration / outcomes /
  kb_name（对应 RAG KB）/ template_book_id（预设模板的存储 key）`。
- `POST /courses/{course_id}/onboarding` — **onboarding 对话轮**。
  - 入参：`{session_id, history:[...], answer?}`。
  - 逻辑：用 `deeptutor.services.llm.complete`（见 `mastery_path.py:246` 的用法）
    驱动一段**轻结构化对话**，围绕四维度（先修基础/学习目标/时间节奏/薄弱点）逐个问；
    维度问全后，产出结构化 `profile`（JSON）并置 `done=true`。
  - 出参：`{message, done, profile?}`。（前端据 `done` 切到"生成/预览"。）
  - Prompt 里明确要求：只问这四个维度、每轮一问、最后输出严格 JSON profile。
    JSON 解析复用 `deeptutor.utils.json_parser.parse_json_response`
    （见 `mastery_path.py:253`）。
- `POST /courses/{course_id}/build-path` — **程序化裁剪 + 注入**。
  - 入参：`{session_id, profile}`。
  - 逻辑：
    1. `LearningStore().load(template_book_id)` 载入该课预设模板（§5 备课产物）。
    2. 按 `profile` **程序化裁剪**（纯 Python，不走 LLM）：
       - 先修基础 → 标记/删除已掌握的入门模块，或设 `current_module_id` 为起点模块；
       - 薄弱点 → 命中的模块/知识点排到靠前、或调高其 `pass_threshold`（重点）；
       - 时间节奏 → 可裁掉"选修/拓展"模块控制长度（用模块命名约定或 §5 里给模块打
         `optional` 标签，通过 extra 字段承载——注意 `KnowledgePoint/LearningModule`
         `model_config = extra="ignore"`，额外标签要放在**模板 sidecar** 或裁剪服务
         自己的映射表里，不要指望存进 LearningProgress）；
       - 学习目标 → 影响详略（首版可仅记录进 profile，供导师 prompt 参考）。
    3. 生成新 `book_id = session_id`（保证 §3.3 的 `resolve_mastery_path_id` 能命中），
       把裁剪后的 modules 通过 **`POST /api/v1/learning/progress/{book_id}/init-modules`**
       写入（或直接 `LearningStore.save`）。
    4. 返回 `{book_id}`，前端据此进预览页。
  - **不改学习引擎**，只组合现有 store/API。

> 为什么不用后端 capability：决策5是"程序化裁剪"而非"LLM 基于模板重生"，所以裁剪是
> 确定性 Python 逻辑，放普通路由/service 最直接、最可控、演示最稳。

**4.2.2 前端：选课页 + onboarding 对话页 + 预览确认页**

   - 数据来自 `GET /api/v1/course/courses`。首版就一门「高等数学」卡片。
   - CTA → 跳 onboarding 页并带 `course_id` + 新建 `session_id`。
   - **不做**顶部自然语言需求输入框（决策15）。

## 5. 备课：把「高等数学」这门课做出来（内容侧，先于演示）

用**原生备课台**（备课模式打开导航）产出两样东西：

1. **课程 RAG 资料 → 一个具名 KB**
   - 进 Knowledge Center（`/knowledge`）新建 KB，名如 `gaodengshuxue`（遵守
     `naming.py` 校验，别用 `/ \ : # %` 等）。
   - 上传/摄入高数教材、讲义、习题（PDF/MD）。记下 KB 名，填进课程目录的 `kb_name`。
2. **预设精通之路模板 JSON**（决策4：自动生成 + 人工微调）
   - 备课模式下开一个 mastery 对话，挂上高数 KB，让导师用 `mastery_build` 依据资料
     生成 modules+知识点（也可用 `POST /api/v1/learning/progress/{tpl}/init-modules`
     手工喂结构）。存储 key 用一个**固定模板 id**，如 `template_gaodengshuxue`。
   - 人工微调：直接编辑 `workspace/learning/template_gaodengshuxue.json`（结构见 §3.1），
     校对模块顺序、`pass_threshold`、每个知识点的 `type`（memory/concept/procedure/design
     选对，直接影响门槛判定）。
   - 把这个模板 id 填进课程目录的 `template_book_id`。
   - **模板绝不要用运行时 session 的 book_id**，避免被学生进度覆盖；裁剪时是"读模板→
     另存为 session book_id"。
3. **课程目录**：先写 `DeepTutor/deeptutor/data/courses.json`（或等价位置），
   一门课一条：`{id:"gaodengshuxue", title:"高等数学", kb_name:"gaodengshuxue",
   template_book_id:"template_gaodengshuxue", difficulty, duration, outcomes:[...],
   summary}`。`GET /courses` 读它。

## 8. 风险 / 注意事项

- **模板与运行进度分离**：模板用固定 id，学生路径用 session_id，裁剪=读模板另存。切勿混。
- **`extra="ignore"`**：`LearningModule/KnowledgePoint` 会丢弃未知字段，"选修/权重"等
  裁剪元数据要放模板 sidecar 或裁剪服务的映射表，别塞进 LearningProgress 指望持久化。
- **book_id 合法性**：session_id 若含 `/ \ : ..` 会被 `storage.py:34`/`mastery_path.py:36`
  拒绝，生成 session_id 时就规避。
- **上游同步**：因是 fork 改造，改动尽量集中在**新增文件**（course_shell 路由、新页面、
  ui-mode），对既有文件（`SidebarShell.tsx`、`main.py`）做**最小侵入**的加法，便于日后
  合并 DeepTutor 上游。
- **两套 capability 概念**别混淆（§3.3），本项目不新增后端 capability。
- **onboarding profile 稳定性**：prompt 要强约束输出严格 JSON + 只问四维度；用
  `parse_json_response` 容错。
- **首版只此一门课**：所有"多课程"扩展（需求输入框、课程网格、配置化）留到后续，
  不在首版扩大范围（决策13/15）。
